# 虚拟机动态内存与已有 VM 兼容

## 功能目标

动态内存基于 libvirt 的 `currentMemory`、`memory` 与 `memballoon` 设备实现：

- `memory` 表示虚拟机可使用的最大内存上限。
- `currentMemory` 表示虚拟机启动时分配的初始内存。
- `memballoon` 允许宿主机在运行中通过 `virsh setmem --live` 调整 VM 当前内存。

本功能默认兼容已有 VM，不会在升级后自动修改已有 VM 的 XML 或运行时内存。管理员明确启用后，才会写入动态内存配置。

新建虚拟机页面中的动态内存默认也是关闭状态，避免创建时自动写入动态内存 metadata；只有管理员在创建或编辑时手动开启后，系统才会按动态内存模式生成配置。

## 兼容模式

后端读取 VM 详情时会优先从 libvirt XML 和 metadata 推断动态内存状态，不依赖数据库保存 VM 层面的内存配置。

旧 VM 没有动态内存 metadata 时，会按静态兼容模式展示：

- `memory_initial = currentMemory`
- `memory_max = memory`
- `memory_min = max(1GB, currentMemory * 50%)`
- `dynamic_enabled = false`
- `memory_compat_mode = legacy_static`

这个推断只用于展示和迁移默认值，不会自动写回 VM XML。

## 管理员启用流程

在 VM 编辑页打开“动态内存配置”：

1. 启用动态内存。
2. 确认启动内存、最小内存、最大内存。
3. 根据需要开启“自动气球调度”。
4. 保存配置。

默认推荐值：

- 启动内存：当前 VM 内存
- 最小内存：当前 VM 内存的 50%，但不低于 1GB
- 最大内存：当前 VM 内存上浮 30%，按 GB 向上取整，例如 2GB 默认推荐 3GB

运行中 VM 保存时只写入“待迁移”metadata，不会修改 live XML，也不会热插入气球设备。下次关机后启动前，后端会自动应用 XML 修改；如果应用失败，会阻止启动并返回中文错误。

已关机 VM 保存时会直接修改 inactive XML，并补齐 `memballoon` 设备。

## Windows 弹性内存（实验）

Windows 虚拟机可以选择“Windows 弹性内存（实验）”模式。该模式基于 QEMU/libvirt 的 `virtio-mem` 设备和 Windows 内部的 `VirtIO Viomem Driver`，用于实现“基础内存 + 可热插拔弹性内存”。

启用后：

- 用户在主表单填写的内存按规格内存理解，不作为最低基础内存。
- 基础内存默认按规格内存的 50% 计算，最低 1GB，并写入 `memory` / `currentMemory`。
- 最大内存默认按规格内存上浮 30% 计算，写入 `maxMemory`，并按差值创建 `virtio-mem` 设备。
- 弹性内存通过 `virsh update-memory-device --requested-size` 调整。
- 初始 `requested` 为 0，后续可按当前内存手动插入额外内存。
- 自动调度器会读取来宾内存统计：使用率超过 70% 时每次扩容 1GB；使用率低于 50% 时计算缩容目标，确保缩容后的使用率不超过 70%。

限制与注意事项：

- 该功能默认关闭，并在界面中标记为实验功能。
- 首次启用或修改基础/最大内存需要关机后应用，不支持直接修改运行中 VM 的基础 XML。
- Windows 内部必须安装 `VirtIO Viomem Driver`。只挂载 `virtio-win.iso` 不会自动安装驱动。
- 缩小弹性内存时，Windows 可能因为当前占用或内存碎片无法立刻全部释放，`requested=0` 后 `current` 仍可能保留一部分。
- 该模式不使用自动气球调度；用户看到的内存规格始终按最大规格理解，基础内存由系统自动计算。
- 生产环境建议先在测试 VM 验证驱动、重启、快照和缩容表现。

## 无气球设备 VM

缺少 `memballoon` 的 VM 在运行中不能启用动态内存迁移。管理员需要安排关机窗口后再启用，后端会在关机状态下补齐气球设备。

这对 aarch64 或历史导入的 VM 尤其重要。运行中热插入气球设备存在兼容风险，当前实现不做自动热插入。

## 自动调度规则

气球调度只有同时满足以下条件的 VM 才会进入自动调度：

- 已启用动态内存。
- 已开启自动气球调度。
- 没有待迁移配置。
- VM 正在运行。
- 能读取到 balloon 内存统计。

Windows 弹性内存使用独立调度策略：

- 已启用动态内存，且内存后端为 `virtio_mem`。
- 没有待迁移配置。
- VM 正在运行。
- 能读取到来宾内存统计。
- 来宾内存使用率超过 70% 时，每次扩容 1GB，不超过最大弹性上限。
- 来宾内存使用率低于 50% 时，按 `ceil(已用内存 / 70%)` 计算缩容目标，不低于基础内存。
- 手动调整当前内存后，会暂停自动调度 10 分钟。

宿主机保护规则：

- 默认保留 `max(2048MB, 宿主机总内存 * 20%)`。
- 增长前检查宿主机 `MemAvailable`，不足则跳过并记录中文警告。
- 只有宿主机内存压力明显时，才允许回收到 `memory_min`。

观察期规则：

- 新启用动态内存后默认 24 小时观察期。
- 观察期内允许自动增长。
- 观察期内不自动回收到启动内存以下。
- 观察期后普通状态仍不低于启动内存，只有宿主机压力明显时才回收到最小内存。

管理员手动调整当前内存后，会暂停该 VM 的自动调度一小段时间，避免人工操作立刻被调度器覆盖。

## 系统设置

系统设置页可以调整以下参数：

- 是否启用动态内存自动调度
- 调度间隔
- 宿主机保留内存 MB
- 宿主机保留比例
- 增长触发阈值
- 回收触发阈值
- 调整冷却时间
- 新启用 VM 的观察期小时数

对应环境变量均以 `KVM_DYNAMIC_MEMORY_` 开头。若环境变量已设置，环境变量优先。

管理员可以在“调度事件”页面查看动态内存调度记录，包括：

- 气球调度
- Windows 弹性内存调度
- 触发原因、执行结果与失败原因

## 配额计算

用户配额仍按启动/保障内存计算：

- 静态 VM：按原有内存值计算。
- 动态 VM：按 `memory_initial` 计算。
- Windows 弹性内存：普通用户主表单填写的是规格内存，创建和编辑配额按该规格内存计算；基础内存只用于 XML 启动值和宿主机初始占用。

最大内存由管理员设置和宿主机保护规则兜底，不直接计入用户基础配额。

## 回滚方式

管理员可以在 VM 编辑页关闭动态内存：

- 后端会写入关闭状态 metadata。
- 不会自动缩小或扩大 VM 当前内存。
- 如需恢复完全静态 XML，建议在关机窗口内手动把最大内存和启动内存调整为一致。

如果待迁移配置导致启动前应用失败，VM 不会进入半配置状态；需要修正动态内存参数或关闭动态内存后再启动。

## Windows 弹性内存报错处理

如果编辑 CPU/内存时报错：

```text
initial memory size of a domain with NUMA nodes cannot be modified with this API
```

说明该虚拟机已经使用 NUMA/virtio-mem 结构，不能再走传统的最大内存修改接口。面板会识别 `virtio_mem` 后端并绕开传统 `setmaxmem` 路径，内存规格变更应通过 Windows 弹性内存配置处理。

首次启用或修改基础内存、最大弹性上限时需要先关机；运行中的 VM 只支持调整当前弹性内存请求值。

## 常用排查命令

```bash
# 查看 VM XML 内存与气球设备
virsh dumpxml VM_NAME --inactive

# 查看运行中 VM balloon 统计
virsh dommemstat VM_NAME

# 查看 VM 当前和最大内存
virsh dominfo VM_NAME

# 手动设置运行中 VM 当前内存，单位 KiB
virsh setmem VM_NAME 4194304 --live
```

如果关闭动态内存创建或启动 VM 时出现 `应用动态内存待迁移配置失败: EOF`，通常是当前 libvirt 在该 VM 没有动态内存 metadata 时返回了空输出。后端会把空输出兼容为“未配置动态内存”，不再阻断普通静态内存 VM 启动。

## 测试建议

局部测试优先覆盖：

- 旧 XML 无 metadata 时能推断静态兼容配置。
- 运行中 VM 启用动态内存只写待迁移状态，不修改 live XML。
- 关机 VM 能写入 `memory`、`currentMemory` 并补齐 `memballoon`。
- 无 `memballoon` VM 在运行中拒绝迁移，并提示需要关机。

生产环境只建议做只读检查。实际迁移应先在测试机验证，再选择低峰关机窗口逐台启用。
