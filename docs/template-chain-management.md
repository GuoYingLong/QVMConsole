# 模板链路管理

## 功能概述

模板现在按“模板族 + 节点”的方式管理：

- 首个模板会生成 `template_uid`，作为整棵模板树的唯一标识
- 每个模板节点会生成 `node_id`
- 从模板创建的虚拟机再次制作模板时，新模板会自动挂到来源模板节点下
- 管理员侧名称用于后台识别，用户侧显示文本用于“从模板克隆”下拉框

模板关系、哈希、类型和凭据均由程序维护，管理员不能手动编辑这些程序元数据。

## 节点字段

模板 `.meta.json` 会记录：

- `template_uid`：模板树唯一标识
- `node_id`：节点唯一标识
- `parent_node_id`：父节点
- `root_node_id`：根节点
- `admin_name`：管理员侧名称
- `display_name`：用户侧显示文本
- `category`：Linux / Windows 模板二级分类；Linux 固定支持 `Ubuntu` / `Debian`，Windows 固定支持 `WindowsServer2022` / `Windows10` / `WindowsServer2012R2`
- `default_config`：模板默认硬件配置
  - `vcpu`：推荐 CPU 核心数
  - `ram`：推荐内存大小（GB）
  - `disk_size`：推荐系统盘大小（GB）
  - `disk_bus`：推荐系统盘总线类型
  - `nic_model`：推荐网卡型号
  - `video_model`：推荐显示设备型号
  - `cpu_topology_mode`：推荐 CPU 拓扑模式
  - `first_boot_reboot_mode`：Windows 首次重启策略
- `clone_visible`：是否允许普通用户克隆
- `disabled`：是否禁用该模板节点
- `md5` / `sha256` / `file_size`：模板磁盘完整性信息

## 启用规则

- 首个模板制作完成后默认启用
- 从模板派生的新节点默认隐藏
- 普通用户只能看到已启用节点
- 管理员可从任意未禁用节点创建虚拟机
- 禁用节点不会出现在普通用户或管理员的新建虚拟机模板下拉框中
- 禁用只影响克隆入口，不会改变模板链路、哈希和导入导出元数据

## 模板树展示

模板管理页按可折叠树展示节点，子节点会挂在来源节点下，便于区分同一基础模板上的多个分支。每个节点会显示“基于哪个上级节点”，并显示：

- 管理员名称和模板文件名
- 用户侧显示文本
- 用户可见状态和禁用状态
- 直接创建的虚拟机数量
- 当前节点子树虚拟机总数
- 哈希记录状态

发布设置中管理员只能维护：

- 管理员名称
- 用户侧显示文本
- 用户可见开关
- 禁用开关
- 默认 CPU / 内存 / 系统盘大小
- 默认系统盘总线类型
- 默认网卡型号

Linux 类型的「制作模板」弹窗中，SSH 初始化凭据按“登录用户名在上、用户密码在下”的顺序展示，便于与后续克隆初始化表单保持一致。

Linux 和 Windows 模板保留“二级分类”字段和分支能力，用于在“从模板创建”下拉框中按 `Linux / Ubuntu`、`Linux / Debian`、`Windows / WindowsServer2022`、`Windows / Windows10`、`Windows / WindowsServer2012R2` 等层级展示：

- 新制作的 Linux 模板默认归入 `Ubuntu`
- 历史 Linux 模板没有分类时会自动补为 `Ubuntu`
- 管理员可以在制作模板或发布设置中选择 `Debian`，但不能输入自定义 Linux 发行版分类
- Debian 13 模板仍属于 `linux` 类型，开通时复用 Linux SSH 初始化流程，和 Ubuntu 一样完成主机名、登录用户、密码与磁盘扩容处理
- 新制作的 Windows 模板默认归入 `WindowsServer2022`
- 历史 Windows 模板会按文件名自动归入 `WindowsServer2022`、`Windows10` 或 `WindowsServer2012R2`
- `WindowsServer2012R2.qcow2` 属于 `windows` 类型和 `WindowsServer2012R2` 分类，克隆时会继续使用模板元数据中的 BIOS、SATA、e1000e、VMVGA 等默认配置

模板名称支持字母、数字、点、下划线和短横线，便于直接使用带版本号的命名方式（如 `ubuntu.24.04`）。出于路径安全考虑，名称中仍禁止出现连续的点 `..`。

## 默认硬件带出

制作模板时，系统会自动从源虚拟机采集以下关键信息并写入模板元数据：

- CPU 核心数
- 内存大小
- 系统盘大小
- 系统盘总线类型
- 首张网卡型号
- 显示设备型号

后续从模板创建或原生链式克隆时，前端会优先使用这些字段作为默认选中值，减少重复填写。Windows 模板会保留源虚拟机的显示设备选择，降低来宾系统自动重启后 VNC 黑屏的概率。

兼容说明：

- 旧模板如果没有 `default_config`，仍可正常使用
- 旧模板会继续沿用当前默认行为，例如从模板磁盘大小回填最小系统盘、网卡/磁盘总线回退到现有默认值

## 删除规则

模板节点删除支持两种模式：

- 级联删除：删除当前节点及其所有子节点。
- 仅删除当前节点并提升子节点：删除中间节点时，直接子模板会通过安全 `qemu-img rebase` 改为继承上级模板，当前节点直接创建的 VM 也会重定向到上级模板。
- 热删除当前节点并提升子节点：运行中的直接 VM 使用 `virsh blockpull` 在线继承上级模板，运行中的子模板后代 VM 使用浅层 `virsh blockcopy --pivot` 在线切换到新 backing。

删除前会展示：

- 即将删除的模板节点链路
- 所有关联虚拟机
- 提升模式下将被提升的直接子模板和需要重定向的直接 VM

级联删除确认后，系统会先删除关联虚拟机，再删除模板磁盘和元数据。

提升模式不会转换为独立硬盘，处理后仍保持链式结构。普通提升模式要求当前节点子树下所有关联 VM 先关机；热提升模式会尽量不关机，若 libvirt block job 或 backing 校验失败，会停止任务并保留当前节点。
