# 模板列表接口性能优化

## 背景

模板列表接口 `/api/template/list` 过去在每次请求时都会执行较重的模板磁盘检测与虚拟机详情枚举，模板或虚拟机数量增加后容易出现明显延迟。

## 当前实现

- 模板引导类型优先复用模板元数据中的 `boot_type`
- 只有在以下场景才会回退检测模板磁盘实际引导类型：
  - 模板元数据缺少 `boot_type`
  - Windows 模板元数据被记录为 `bios`，需要纠正为实际值
- 模板关联虚拟机数量统计改为直接扫描 `/etc/libvirt/qemu/*.xml` 中的模板来源元数据，不再为列表页调用全量 `virsh` 详情查询
- 模板磁盘大小信息改为优先复用单次 `qemu-img info --output=json` 的 `actual-size` / `virtual-size`，不再为每个模板额外执行一次 `du`
- 模板磁盘大小信息按文件 `size + mtime` 做进程内缓存；同一模板文件未变化时，后续请求直接复用缓存结果
- 首次批量加载模板时会并发补齐磁盘信息，减少模板数量多时的串行等待时间
- Windows 模板的 `boot_type=bios` 现在会在首次探测后写入 `boot_verified=true`，避免 BIOS 模板每次列表请求都重复执行 `virt-filesystems`

## 效果

在测试机当前模板数据下，`/api/template/list` 的响应时间已从约 9 秒降到约 0.6 秒；当模板数量继续增长时，重复请求会进一步受益于磁盘信息缓存，避免再次为每个模板重复执行外部命令。
