# 桥接直通网桥使用文档

## 功能概述

管理员可在「网络」-「网络概览」中查看宿主机物理网卡、OVS 网桥和交换机使用情况，并创建桥接直通网桥。

桥接直通网桥会把虚拟机网卡接入指定 OVS 网桥，网桥再上联到物理网卡。接入该网桥的虚拟机网络由上级路由器分配地址，不使用面板内部 DHCP、NAT、端口转发或安全组。

## 网桥类型

| 类型 | 说明 |
|------|------|
| 内网 NAT | 默认 `br-ovs`，继续使用面板 DHCP、NAT、VPC VLAN、安全组和端口转发 |
| 桥接直通 | 管理员创建的新网桥，直接桥接到物理网卡，由上级路由器分配 IP |

## 创建桥接网桥

在「网络概览」点击「创建桥接网桥」，填写：

| 字段 | 说明 |
|------|------|
| 网桥名称 | Linux 接口名最长 15 个字符，例如 `brpub0` |
| 物理网卡 | 要桥接的宿主机物理网卡 |
| 迁移宿主机 IP | 若该物理网卡承载宿主机管理 IP 或默认路由，通常需要开启 |

创建前后系统会写入 `/etc/kvm-console/bridges/<bridge>.sh` 和 `kvm-console-bridges.service`，用于宿主机重启后恢复面板管理的桥接网桥。若开启「迁移宿主机 IP」，恢复脚本会在物理口加入 OVS 前记录 IPv4 地址、默认网关和路由 metric，加入后把这些配置迁移到 OVS bridge，避免物理口已经成为 OVS 端口但仍持有宿主机管理 IP 导致网关 ARP 失败。

创建桥接网桥导致默认出口切换时，后端会按当前默认出口重建 `br-ovs` 与 NAT 模式 VPC 交换机的 NAT/FORWARD 规则，并清理仍指向旧出口的残留规则。

## 创建桥接交换机

管理员创建交换机时可选择目标网桥：

- 选择 `br-ovs`：行为与现有 VPC 交换机一致。
- 选择桥接直通网桥：交换机不创建内部网关口、不启动 dnsmasq、不写 NAT 和 ACL。
- 桥接直通交换机可选填“桥接 VLAN ID”：`0` 表示不打 VLAN；填写 `1-4094` 时，VM 接入该桥接交换机的 OVS 端口会使用对应 VLAN tag，上级网络需要允许该 VLAN。
- 桥接直通交换机支持三项安全策略：
  - 混杂模式：默认拒绝，系统会对 VM 端口启用 `no-flood`，减少未知单播泛洪到 VM。
  - MAC 地址更改：默认拒绝，VM 只能使用 XML 中配置的 MAC 作为源 MAC 发包。
  - 伪传输：默认拒绝，源 MAC 与配置 MAC 不一致的发包会被 OVS 丢弃。

桥接直通交换机仍会保留交换机级月流量统计和聚合带宽限制。流量统计基于 VM 网卡历史统计；带宽限制基于 OVS meter 应用到该交换机绑定 VM 的运行态端口和目标 MAC。

若历史 VM 已经接入桥接网桥但缺少交换机绑定记录，系统会在桥接网桥只有一个桥接直通交换机时，从运行态 `domiflist` 和持久化 XML 识别该 VM，并参与交换机流量统计和 OVS meter 限速。桥接直通下行限速按 VM MAC 匹配所有发往该 VM 的流量，不依赖内部 CIDR 或单一上联端口。

## 虚拟机接入

管理员创建、克隆或导入 VM 时，如果选择桥接直通交换机：

- VM XML 会直接接入目标 OVS 网桥。
- VM 内部需要从上级路由器获取 IP。
- 安全组选择会被忽略。
- 端口转发不适用于桥接直通 VM。
- VM 详情页会隐藏端口转发和静态 IP 管理入口。

已有 VM 切换到或切出桥接直通交换机时，建议先关机再操作；运行中的 VM 若不在目标网桥上，后端会拒绝切换，避免断网。

从桥接直通切回普通 VPC 时，系统会把 VM XML 的 OVS source bridge 恢复为 `br-ovs`，并写入目标交换机 VLAN。若 VM 正在运行且仍挂在桥接直通网桥上，后端会要求先关闭虚拟机，避免数据库已经切换但运行时端口仍留在桥接网桥上。

删除开启「迁移宿主机 IP」的桥接网桥时，后端会先删除对应开机恢复脚本，再把 bridge 上的 IPv4、默认网关和路由 metric 迁回物理网卡，最后删除 OVS 端口和网桥。删除后会按当前默认出口重建 `br-ovs` 与 NAT 模式 VPC 交换机的 NAT/FORWARD 规则，并清理仍指向旧桥接出口的残留规则。若已没有其它桥接恢复脚本，系统会禁用 `kvm-console-bridges.service`。

## 测试机验证记录

测试机已验证使用 `ens33` 临时接入短名 OVS 网桥 `brt0`：

- 宿主机 IP `192.168.11.18/24` 可迁移到 `brt0`。
- 默认路由可切换为 `dev brt0`。
- 上级网关 `192.168.11.2` 可 ping 通。
- 回滚后 `ens33` 恢复 IP 和默认路由，临时网桥已删除。

公网 ICMP 在测试机当前网络下不通，因此连通性以默认网关和 SSH 未中断为准。

## 故障排查

如果宿主机重启后出现 SSH 超时，但 VNC 内 `ens33` 仍显示宿主机 IP，并且 `ip neigh show dev ens33` 中网关为 `FAILED`，通常表示物理口被恢复脚本加入 OVS bridge 后，宿主机 IP 没有迁移到 bridge。可先通过 VNC 临时恢复：

```bash
ovs-vsctl --if-exists del-port <bridge> <uplink>
ip link set <uplink> up
ping -c 4 <gateway>
```

恢复后检查 `/etc/kvm-console/bridges/<bridge>.sh` 是否包含 `HOST_ADDRS`、`ip addr flush dev "$UPLINK"` 和 `ip route replace default via "$HOST_GW" dev "$BRIDGE"` 等迁移逻辑。
