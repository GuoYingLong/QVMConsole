# 端口转发 HTTP 建站探测与自动封禁

## 功能说明

系统会对 `TCP` 端口转发做明文 HTTP 探测：

- 后台定时任务默认每 `60` 分钟执行一次
- 管理员可在「网络中心 → 端口转发」手动触发全局探测
- 管理员或虚拟机归属用户可在虚拟机详情页的「网络管理 → 端口转发」手动触发当前 VM 探测

探测直接访问转发目标 `目标 IP:目标端口`，发送最小 HTTP 请求：

- 只判断是否存在明文 `HTTP` 响应
- 只要成功读到 HTTP 状态行并解析出状态码，就视为存在建站行为
- `HTTPS/TLS` 不在本次探测范围内

## 自动封禁规则

当满足以下条件时，系统会自动封禁对应转发：

1. 转发协议为 `TCP`
2. 探测到明文 HTTP 响应
3. 该转发不命中白名单

封禁动作说明：

- 会删除当前生效的 `iptables` 端口转发规则
- 会保留封禁快照，方便在列表中继续展示状态和原因
- 鼠标悬停状态标签可查看具体封禁原因

## 白名单规则

白名单由管理员统一在「网络中心 → 端口转发」维护，分为两类：

- 用户白名单
- 虚拟机白名单

白名单录入方式：

- 用户白名单使用可输入搜索的下拉框选择用户
- 虚拟机白名单使用可输入搜索的下拉框选择虚拟机

额外规则：

- 管理员归属的转发，或由管理员创建的转发，默认视为白名单
- 普通用户页面不提供白名单编辑入口
- 普通用户若当前用户和当前 VM 都未命中白名单，虚拟机详情页会显示风险横幅

## 白名单恢复

新增白名单后，系统会立即尝试恢复对应范围内已被自动封禁的转发：

- 新增用户白名单：恢复该用户下已封禁的转发
- 新增虚拟机白名单：恢复该 VM 下已封禁的转发

如果同范围内已经存在“状态正常但安全组未补齐”的历史规则，重新保存对应白名单时也会自动补齐安全组放行。

如果恢复失败：

- 封禁记录会保留
- 接口返回会附带失败说明

## 页面状态说明

端口转发表状态统一显示为：

- `正常`：当前转发处于可用状态
- `封禁`：当前转发因探测命中 HTTP 且未报备而被封禁

说明：

- 只有 `封禁` 状态支持鼠标悬停查看封禁原因
- `正常` 状态不显示悬停提示

## 相关接口

### 白名单

- `GET /api/network/port-forward/whitelist`
- `POST /api/network/port-forward/whitelist/user`
- `DELETE /api/network/port-forward/whitelist/user/:username`
- `POST /api/network/port-forward/whitelist/vm`
- `DELETE /api/network/port-forward/whitelist/vm/:vm_name`
- `GET /api/network/port-forward/whitelist/summary?vm_name=...`

### 手动探测

- `POST /api/network/port-forward/probe/run`

请求体：

```json
{
  "vm_name": "vm-demo"
}
```

说明：

- 不传 `vm_name` 表示全局探测，仅管理员可用
- 传 `vm_name` 表示只探测当前虚拟机，管理员和归属用户可用

### 删除已封禁记录

- `DELETE /api/network/port-forward/by-key/:rule_key`

说明：

- 仅用于删除已经封禁、当前不再 live 的快照记录
- 正在生效的端口转发仍使用原有按 `id` 删除接口
