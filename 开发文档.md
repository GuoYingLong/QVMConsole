# 容器云功能开发文档

## 概述

在侧边栏新增「容器云」功能模块，基于 Docker Engine API 实现容器化管理，与现有 KVM 虚拟机管理面板并行运行。所有容器使用宿主机已安装的 Docker，通过 Go 后端调用 Docker SDK 或 CLI 完成操作。

---

## 页面结构

侧边栏新增一级菜单「容器云」，下设 5 个标签页（Tabs），统一使用类似虚拟机列表的标签结构。

| 标签页 | 功能说明 |
|--------|----------|
| **容器** | 展示所有容器信息（名称、状态、镜像、CPU、内存、磁盘、网络、端口映射、创建时间），支持创建/启动/停止/重启/删除/查看详情 |
| **编排** | 展示所有 Compose 编排信息（编排名称、服务数、状态），支持新建/上传 YAML/启动/停止/删除 |
| **镜像** | 展示所有镜像信息（仓库、标签、大小、创建时间），支持拉取/删除/导入/导出 |
| **卷** | 展示所有卷信息（名称、驱动、大小、挂载点、关联容器），支持创建/删除/详情 |
| **网络** | 展示所有网络信息（名称、驱动、子网、网关、连接的容器数），支持创建/删除/详情 |

## 重要约束

### 1. 存储约束
- **所有容器持久化数据**必须存储在用户的`我的存储`路径下
- 存储路径结构：`/var/lib/kvm-user-storage/<用户名>/containers/<容器名或编排名>/`
- `containers` 目录作为`我的存储`新增的子类别，需在初始化存储池时自动创建
- 容器目录纳入用户已有的 Project Quota，与 ISO/Share/Disk 合并计算
- 在「我的存储」页面新增**「容器存储」标签页**，展示所有容器的持久化文件（详见下方「我的存储 - 容器存储标签页」章节）

### 1.1 端口映射约束
- 容器的宿主端口映射**必须使用系统端口范围**（`AutoPortStart` ~ `AutoPortEnd`，与虚拟机端口转发共享同一端口池）
- 创建容器时，用户只需填写容器端口，宿主端口**由系统自动从空闲端口中分配**
- 用户也可手动指定宿主端口，但必须校验在系统端口范围内且未被占用
- 编排文件中的 `ports` 配置同样**强制改写宿主端口**为系统自动分配的端口（详见编排管理章节）

### 2. 资源配额约束
- 容器资源**纳入用户总配额体系**，与虚拟机共享配额池
- 每个容器创建时必须限制以下资源：
  - **CPU**：`--cpus` 限制（计入用户 `max_cpu` 配额，单位：核心数）
  - **内存**：`--memory` 限制（计入用户 `max_memory` 配额，单位：GB）
  - **磁盘**（持久化卷）：卷大小计入用户 `max_storage` 存储配额
  - **网络限速**：复用现有带宽配额体系（`max_bandwidth_up/down`），与虚拟机共享带宽池，使用 `tc`（traffic control）对容器 veth 网卡进行限速
- **非持久化存储上限**：由于服务器文件系统为 ext4（`--storage-opt` 需要 xfs+pquota），改为**后台监控 + 启动校验**方式限制容器可写层
  - 所有容器强制应用日志限制：`--log-opt max-size=50m --log-opt max-file=3`
  - **后台定时检查**：每 60 秒通过 `docker inspect -s` 获取容器 `SizeRw`，超过 **1GB** 时**强制停止容器**
  - **启动校验**：容器启动前检查 `SizeRw`，超过 1GB 则**拒绝启动**，提示用户必须删除该容器后重新创建
  - 持久化数据全部挂载到用户存储目录，由 Project Quota 统一管控
- 配额检查时机：
  - 创建容器/编排时检查 CPU + 内存 + 磁盘
  - 启动容器时再次校验 CPU + 内存（类似虚拟机开机检查）
  - 启动容器时自动应用网络限速（基于用户带宽配额均分）
  - 修改容器资源不触发配额检查，仅在启动时校验

### 3. 用户存储前置校验
- 操作容器功能前，需校验用户是否已**开通存储**（`IsStorageInitialized()`）
- 未开通存储的用户，显示引导提示，链接到「我的存储」开通

### 4. 操作权限
- **管理员**：可管理所有容器，不受配额限制
- **普通用户**：只能操作自己的容器，受配额限制
- 容器归属通过 Docker Label 标记：`kvm.owner=<用户名>`

## 详细功能设计

### 一、容器管理

#### 1.1 容器列表

| 字段 | 来源 | 说明 |
|------|------|------|
| 容器名称 | `docker inspect` | - |
| 状态 | `docker inspect` → State.Status | running / stopped / paused / restarted |
| 镜像 | `docker inspect` → Config.Image | - |
| CPU 限制 | `docker inspect` → HostConfig.NanoCpus | 换算为核心数显示 |
| 内存限制 | `docker inspect` → HostConfig.Memory | 换算为 MB/GB 显示 |
| 网络限速 | `tc` 规则读取 | 显示下行/上行平峰速率 (Mbps) |
| 端口映射 | `docker inspect` → NetworkSettings.Ports | 如 `0.0.0.0:8080->80/tcp` |
| 创建时间 | `docker inspect` → Created | - |
| 运行时长 | `docker inspect` → State.StartedAt | 计算运行时长 |

- 列表使用 SSE 实时推送状态更新（复用现有 SSE 模式）
- 支持操作按钮：启动、停止、重启、暂停/恢复、删除、查看日志、进入终端

#### 1.2 创建容器

分步表单（类似虚拟机创建向导）：

**步骤 1：镜像选择**
- 从已有镜像列表中选择
- 或输入镜像名称从远程拉取（如 `nginx:latest`）
- 显示镜像标签、大小

**步骤 2：资源配置**
- **容器名称**：必填，自动校验唯一性
- **CPU 限制**：必填，下拉/输入框，单位：核心，支持 0.5 等小数
- **内存限制**：必填，下拉/输入框，单位：MB，最小 64MB
- 实时显示用户剩余配额

**步骤 3：网络配置**
- 选择网络模式：bridge（默认）/ host / none / 已有自定义网络
- 端口映射配置：可动态添加多条规则
  - 用户只需填写**容器端口**和**协议**（tcp/udp）
  - **宿主端口由系统自动分配**（从 `AutoPortStart` ~ `AutoPortEnd` 范围内选取空闲端口，与虚拟机端口转发共享端口池）
  - 用户也可手动指定宿主端口，但后端校验必须在系统端口范围内且未被占用
  - 容器端口映射通过 Docker 自身的 `-p` 参数实现（Docker 管理宿主机端口监听），无需额外写 iptables DNAT
  - 分配后的宿主端口需在 UFW 中放行
- 容器 IP（可选）：对于自定义网络可指定固定 IP
- **速率限制**（可选）：
  - 下行速率（Mbps）：该容器的下行平峰速率，0=由系统自动均分
  - 上行速率（Mbps）：该容器的上行平峰速率，0=由系统自动均分
  - 留空/设0时，启动后自动使用均分值（用户带宽配额 ÷ 运行中的 VM+容器数量）

**步骤 4：存储配置**
- 持久化卷映射：可动态添加多条
  - 自动将宿主路径设置为用户存储目录下的子路径
  - 格式：`/var/lib/kvm-user-storage/<用户名>/containers/<容器名>/<目录名>:<容器内路径>`
  - 读写模式选择（rw/ro）
- 显示当前存储使用量和剩余配额

**步骤 5：高级配置**
- 环境变量：动态添加 KEY=VALUE 对
- 重启策略：no / always / on-failure / unless-stopped
- 命令覆盖（CMD）：可选
- 入口点覆盖（Entrypoint）：可选
- 特权模式：开关（默认关闭，仅管理员可开启）

#### 1.3 容器详情页

标签页结构（类似虚拟机详情页）：

| 标签页 | 内容 |
|--------|------|
| **基本信息** | 容器 ID、名称、镜像、状态、创建时间、运行时长、资源限制、网络配置、卷挂载列表 |
| **日志** | 实时容器日志流（`docker logs -f`），支持下载日志、搜索、行数限制 |
| **监控** | CPU/内存/网络 IO/磁盘 IO 实时图表（`docker stats`），使用 SSE 推送 |
| **事件** | 容器生命周期事件列表（start/stop/die/restart 等） |
| **终端** | WebSocket 终端（`docker exec -it`），参考现有 VNC WebSocket 实现 |
| **操作** | 启动/停止/重启/暂停/恢复/删除/修改资源配置/修改重启策略/修改速率限制 |

### 二、编排管理

#### 2.1 编排列表

| 字段 | 来源 | 说明 |
|------|------|------|
| 编排名称 | `docker compose ls` | 编排项目名 |
| 服务数量 | `docker compose ps` | 该编排下的服务总数 |
| 运行状态 | `docker compose ps` | running(x)/exited(y) |
| 配置文件 | 数据库记录 | YAML 文件路径 |

#### 2.2 创建/上传编排

1. **新建**：提供 YAML 编辑器（Monaco Editor / CodeMirror），编写 docker-compose.yml
2. **上传**：上传 `.yml` / `.yaml` 文件

**路径与端口校验与改写（关键逻辑）：**

解析编排文件内容时，后端必须对 `volumes`（持久化路径）和 `ports`（端口映射）两部分进行强制改写：

```yaml
# 用户原始编排
services:
  db:
    image: mysql:8
    ports:
      - "3306:3306"
    volumes:
      - /opt/data/mysql:/var/lib/mysql
      - ./config:/etc/mysql/conf.d
  web:
    image: nginx:latest
    ports:
      - "80:80"
      - "443:443"

# 改写后（用户名为 user1，编排名称为 myapp，系统端口范围 10000-60000）
services:
  db:
    image: mysql:8
    ports:
      - "10042:3306"    # 宿主端口自动分配为 10042
    volumes:
      - /var/lib/kvm-user-storage/user1/containers/myapp/mysql:/var/lib/mysql
      - /var/lib/kvm-user-storage/user1/containers/myapp/config:/etc/mysql/conf.d
  web:
    image: nginx:latest
    ports:
      - "10043:80"      # 宿主端口自动分配为 10043
      - "10044:443"     # 宿主端口自动分配为 10044
```

**volumes 改写规则：**
- 提取绑定挂载（bind mount）中的宿主路径部分
- 将宿主路径替换为 `<用户存储路径>/containers/<编排名称>/<最后一级目录名>` 
- 相对路径（`./xxx`）也需改写
- 命名卷（named volume）不需要改写，但需要添加 `kvm.owner` Label
- 改写后保存到用户容器存储目录，并记录原始文件和改写后文件的映射

**ports 改写规则：**
- 解析所有 `ports` 条目中的宿主端口
- **无论用户写了什么宿主端口，一律替换为系统从 `AutoPortStart` ~ `AutoPortEnd` 范围内自动分配的空闲端口**
- 分配时需要排除已被虚拟机端口转发和其他容器占用的端口
- 分配后的端口映射关系记录到数据库（Compose 模型中），便于查看和管理
- 编排的端口映射同样需要在 UFW 中放行

#### 2.3 编排操作

- **启动**：`docker compose -f <file> up -d`
- **停止**：`docker compose -f <file> down`
- **重启**：`docker compose -f <file> restart`
- **删除**：停止所有服务 + 删除编排 YAML + 清理持久化目录（需确认）
- **查看日志**：汇总或按服务分别查看
- **编辑**：重新编辑 YAML，保存后自动重新路径校验

### 为什么编排需要数据库

编排需要在数据库中记录以下信息，因为这些信息无法仅从 Docker CLI 获取：

- 编排文件的原始内容（用户上传/编写的原始 YAML）
- 改写后的最终 YAML 路径
- 编排所属用户（虽然也可以用 Label，但编排启停需要知道文件位置）
- 编排的存储路径映射（方便管理和清理）

```go
// Compose 编排记录
type Compose struct {
    ID           uint           `json:"id" gorm:"primaryKey"`
    Name         string         `json:"name" gorm:"uniqueIndex;size:100;not null"`  // 编排名称
    Username     string         `json:"username" gorm:"size:64;not null;index"`      // 所属用户
    OriginalYAML string         `json:"original_yaml" gorm:"type:text"`              // 原始 YAML 内容
    FilePath     string         `json:"file_path" gorm:"size:500"`                   // 改写后文件路径
    StorageDir   string         `json:"storage_dir" gorm:"size:500"`                 // 持久化存储目录
    Status       string         `json:"status" gorm:"size:20;default:'stopped'"`     // 运行状态
    CreatedAt    time.Time      `json:"created_at"`
    UpdatedAt    time.Time      `json:"updated_at"`
    DeletedAt    gorm.DeletedAt `json:"-" gorm:"index"`
}
```

### 配额集成方案

在现有 `service/user_quota.go` 中扩展，增加容器资源统计：

```go
// 计算用户已用资源（包含容器）
func CalculateUserUsage(username string) QuotaUsage {
    vmUsage := calculateVMUsage(username)
    containerUsage := calculateContainerUsage(username) // 新增

    return QuotaUsage{
        UsedCPU:    vmUsage.CPU + containerUsage.CPU,
        UsedMemory: vmUsage.Memory + containerUsage.Memory,
        UsedDisk:   vmUsage.Disk + containerUsage.Disk, // 容器卷大小
        UsedVM:     vmUsage.Count,                       // VM数量不含容器
        UsedContainer: containerUsage.Count,              // 容器数量单独统计
    }
}
```

需要在用户模型中新增容器配额字段：

```go
// 在 model/user.go 中新增
MaxContainer int `json:"max_container" gorm:"default:0"` // 最大容器数量，0=不限
```

### 用户存储扩展

在现有 `service/user_storage.go` 中新增容器存储目录：

```go
const StorageCategoryContainer = "containers" // 容器持久化存储

// GetUserContainerDir 获取用户容器存储路径
func GetUserContainerDir(username string) string {
    return fmt.Sprintf("%s/%s/containers", GetStorageMountPoint(), username)
```

### 容器网络限速方案

#### 设计原则

与虚拟机带宽限速共享同一套配额体系，用户的 `max_bandwidth_up/down` 配额由其名下所有 **VM + 容器** 共同分摊。

#### 技术实现

虚拟机使用 `virsh domiftune` + `tc` 限速，容器则完全使用 `tc`（traffic control）对容器的 **veth** 网卡进行限速：

```
宿主机 veth 网卡方向说明（从宿主机视角）：
- egress（发出）= 数据发往容器 = 容器的下行
- ingress（接收）= 数据来自容器 = 容器的上行

tc 只能直接限制 egress，ingress 需通过 ifb 设备或 ingress qdisc 实现
```

**限速方式：**

1. **获取容器 veth 网卡**：通过 `docker inspect` 获取容器 PID，在容器网络 namespace 中使用 `ip link show eth0` 解析 `@ifN` 获取 peer ifindex，再在宿主机上反查 veth 接口名（经测试比 sysfs 方式更可靠）
2. **下行限速（egress on veth）**：在宿主机侧 veth 的 egress 方向使用 `tc htb` 设置速率
3. **上行限速（ingress on veth）**：在宿主机侧 veth 创建 `ingress qdisc` 使用 `tc filter` 限速

```go
// ApplyContainerBandwidth 使用 tc 命令限制容器网络速率
// 参数单位：KB/s（与 VM 限速保持一致）
func ApplyContainerBandwidth(containerName string, downAvgKBps, upAvgKBps int) error {
    // 1. 获取容器的 veth 接口名
    vethIF := getContainerVeth(containerName)
    if vethIF == "" {
        return fmt.Errorf("无法获取容器 %s 的 veth 接口", containerName)
    }

    // 2. 清除已有规则
    utils.ExecShell(fmt.Sprintf("tc qdisc del dev %s root 2>/dev/null", vethIF))
    utils.ExecShell(fmt.Sprintf("tc qdisc del dev %s ingress 2>/dev/null", vethIF))

    // 3. 下行限速（veth egress = 容器下行）
    if downAvgKBps > 0 {
        rateKbit := downAvgKBps * 8
        burstBytes := downAvgKBps * 1024 / 10
        if burstBytes < 15360 { burstBytes = 15360 }
        utils.ExecShell(fmt.Sprintf(
            "tc qdisc add dev %s root handle 1: htb default 1", vethIF))
        utils.ExecShell(fmt.Sprintf(
            "tc class add dev %s parent 1: classid 1:1 htb rate %dkbit ceil %dkbit burst %d",
            vethIF, rateKbit, rateKbit, burstBytes))
    }

    // 4. 上行限速（veth ingress = 容器上行）
    if upAvgKBps > 0 {
        rateKbit := upAvgKBps * 8
        utils.ExecShell(fmt.Sprintf(
            "tc qdisc add dev %s handle ffff: ingress", vethIF))
        utils.ExecShell(fmt.Sprintf(
            "tc filter add dev %s parent ffff: protocol ip u32 match u32 0 0 police rate %dkbit burst 15k drop flowid :1",
            vethIF, rateKbit))
    }

    return nil
}

// getContainerVeth 获取容器的宿主机侧 veth 接口名
// 使用 ip link show 解析 @ifN 方式获取 peer ifindex（比 sysfs 更可靠）
func getContainerVeth(containerName string) string {
    // 获取容器 PID
    result := utils.ExecShell(fmt.Sprintf(
        "docker inspect -f '{{.State.Pid}}' '%s'", containerName))
    pid := strings.TrimSpace(result.Stdout)
    if pid == "" || pid == "0" { return "" }

    // 在容器网络 namespace 中通过 ip link show eth0 获取 peer ifindex
    // 输出格式: "2: eth0@if29: <BROADCAST,...>" → 提取 29
    result = utils.ExecShell(fmt.Sprintf(
        "nsenter -t %s -n ip link show eth0 | head -1 | sed 's/.*@if//' | sed 's/:.*//'", pid))
    peerIfindex := strings.TrimSpace(result.Stdout)
    if peerIfindex == "" { return "" }

    // 在宿主机上根据 ifindex 找到对应的 veth 接口
    result = utils.ExecShell(fmt.Sprintf(
        "ip -o link | grep '^%s:' | awk '{print $2}' | tr -d ':' | cut -d'@' -f1", peerIfindex))
    return strings.TrimSpace(result.Stdout)
}
```
